# ALevelPack Format

## Purpose

`alevelpack` - это authoring/storage format для редактора уровней `Arrows`.

Это:

- не runtime-формат игры;
- не внутренний state редактора;
- внешний формат хранения одного или нескольких уровней.

## Why This Format

Выбрали свой контейнерный формат, потому что нам одновременно нужно:

- хранить поле как цветовую карту;
- хранить стартовые точки отдельно;
- уметь хранить несколько уровней;
- иметь один файл снаружи, а не пару разрозненных файлов;
- оставить путь к будущему расширению формата без переделки базовой идеи.

## ALevelPack v1

### Outer Format

`alevelpack` v1 - это один контейнерный файл с собственным расширением.

Практическая реализация:

- обычный zip-контейнер;
- собственное расширение файла, например `main.alevelpack`.

Это дает:

- один файл для пользователя;
- простую упаковку/распаковку;
- возможность хранить несколько уровней;
- возможность расширять формат без перехода на сырой бинарный парсер.

### Internal Structure

Предлагаемая структура контейнера:

```text
manifest.json
palette.json
levels/
  20260605_213210_456_a7k3/
    board.png
    meta.json
  20260605_213415_102_m2qd/
    board.png
    meta.json
```

### manifest.json

Содержит:

- идентификатор формата;
- версию;
- имя пака;
- `lastOpenedLevelId`;
- список уровней;
- порядок уровней.

Черновой пример:

```json
{
  "format": "arrows-level-pack",
  "version": 1,
  "name": "main-pack",
  "lastOpenedLevelId": "20260605_213210_456_a7k3",
  "levels": [
    { "id": "20260605_213210_456_a7k3", "path": "levels/20260605_213210_456_a7k3" },
    { "id": "20260605_213415_102_m2qd", "path": "levels/20260605_213415_102_m2qd" }
  ]
}
```

### Level Identifiers

У каждого уровня должен быть свой уникальный `id`.

Текущее правило генерации `level id`:

- без префикса `level`;
- формат: `yyyyMMdd_HHmmss_SSS_xxxx`
- где:
  - `yyyyMMdd_HHmmss_SSS` — локальные дата и время с миллисекундами;
  - `xxxx` — короткий случайный суффикс.

Пример:

- `20260605_213210_456_a7k3`

Зачем так:

- id остается читаемым глазами;
- id почти наверняка уникален;
- этого достаточно для редактора, где уровни создает один пользователь без параллельной генерации.

Требование:

- `id` используется как логический идентификатор уровня;
- тот же `id` используется в пути папки уровня внутри контейнера.

### palette.json

Содержит:

- список допустимых цветов;
- зарезервированный цвет для inactive.

Черновой пример:

```json
{
  "inactiveColor": "#FFFFFF",
  "colors": [
    "#D32F2F",
    "#E53935",
    "#F06292"
  ]
}
```

### board.png

Это основная карта уровня.

Правила:

- размер изображения = размер поля;
- `1 пиксель = 1 клетка`;
- цвет пикселя = состояние клетки;
- обычные цвета = части линий;
- специальный белый цвет = `inactive`;
- пустых клеток в сохраненном уровне быть не должно.

То есть после сохранения весь board должен быть полностью заполнен пикселями допустимых цветов.

### meta.json

Содержит данные, которые не должны кодироваться цветом.

На первом этапе:

- `startPoints`
- `checked`

Черновой пример:

```json
{
  "checked": true,
  "startPoints": [
    { "x": 2, "y": 5, "direction": "right" },
    { "x": 8, "y": 1, "direction": "down" }
  ]
}
```

### startPoints

Стартовые точки теперь храним не только как координаты, но и с явным направлением.

Формат одной стартовой точки:

- `x`
- `y`
- `direction`

Допустимые значения `direction`:

- `left`
- `right`
- `up`
- `down`

Черновой пример:

```json
{
  "x": 2,
  "y": 5,
  "direction": "right"
}
```

Смысл:

- `start` больше не является просто меткой клетки;
- у старта есть явно сохраненное направление выхода линии;
- это направление является частью данных уровня и должно загружаться/сохраняться как есть.

Правило допустимого направления:

- `right` допустимо, если слева есть соседняя клетка того же цвета;
- `left` допустимо, если справа есть соседняя клетка того же цвета;
- `up` допустимо, если снизу есть соседняя клетка того же цвета;
- `down` допустимо, если сверху есть соседняя клетка того же цвета.

То есть сохраненное направление старта должно соответствовать реальной геометрии линии за стартовой клеткой.

### checked

`checked` - это сохраненный статус того, что текущая версия уровня успешно прошла ручную проверку через `Check Level`.

Правила:

- `checked: true` означает, что именно текущее сохраненное состояние уровня прошло проверки;
- `checked: false` означает, что уровень либо еще не проверяли, либо после проверки он был изменен;
- `checked` хранится в `meta.json` каждого уровня;
- при загрузке pack-файла редактор обязан восстанавливать checked-статус уровня из файла.

Важно:

- `checked` не вычисляется заново при обычном открытии файла;
- `checked` меняется только в результате:
  - успешного `Check Level`
  - или последующего редактирования уровня, которое должно сбрасывать checked обратно в `false`.

## Data Semantics

### Field Encoding

- любой обычный цвет = обычная игровая клетка / часть линии;
- `#FFFFFF` = `inactive`;
- “пустой клетки” в сохраненном уровне не существует.

### Start Points

Стартовые точки храним отдельно от bitmap.

Причина:

- не хотим смешивать цвет клетки и служебный смысл старта;
- не хотим заводить специальные “стартовые оттенки” цветов;
- стартовые точки sparse по природе, для них metadata естественнее.

При этом стартовая точка теперь содержит:

- координаты;
- явное направление.

То есть направление старта больше не обязано только вычисляться из соседней геометрии.

### Checked Status

Статус checked тоже храним отдельно от bitmap.

Причина:

- это не свойство клетки и не свойство цветовой карты;
- это служебный статус редактора/проверки уровня;
- его естественное место - metadata уровня.

### Multi-Level Storage

В одном `alevelpack` можно хранить несколько уровней.

Это пригодится для:

- набора уровней;
- тестовых паков;
- редакторной работы пачками;
- будущей группировки уровней по темам или мирам.
