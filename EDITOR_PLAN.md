# Arrows Level Editor - Plan v0.1

## Goal

Сначала делаем удобный визуальный редактор уровня.
Формат сохранения не фиксируем заранее: сначала нужно понять, что именно мы рисуем, какие режимы реально удобны, и какие данные рождаются естественно из редактора.

Приоритет:

1. Быстро создать поле нужного размера.
2. Удобно рисовать линии цветами.
3. Удобно отмечать неактивные клетки.
4. Удобно видеть начало линии.
5. Уже после этого решить, как лучше сохранять результат.

## Main Product Decision

Первый шаг - не runtime и не формат уровня, а standalone editor-first prototype.

Технологическое решение для первого прототипа:

- делаем редактор на `Flutter`;
- целимся в запуск на macOS и Windows;
- дополнительно оставляем возможность позже запускать тот же редактор на телефоне;
- `libGDX` сейчас не берем как основной стек редактора, потому что на этом этапе важнее скорость сборки tool-like UI, чем близость к игровому runtime.

Редактор должен быть похож на простой paint-like grid editor:

- есть крупная квадратная сетка;
- есть размеры поля;
- есть кнопка генерации поля;
- есть палитра цветов;
- есть color picker для добавления или переопределения цветов;
- есть режим рисования линий;
- есть режим пометки неактивных клеток;
- есть режим выбора/редактирования;
- есть видимый маркер начала линии.

## Editor V1 Scope

### 1. Field Generation

Пользователь задает размер поля числами:

- width
- height

Примеры:

- 10x10
- 20x20
- любой произвольный размер в разумных пределах

Рядом с полями размеров должна быть кнопка:

- `Generate`

Поведение:

- при вводе новых чисел поле само не ломается мгновенно;
- по нажатию `Generate` создается или пересоздается новое поле;
- сетка визуально состоит из больших квадратных клеток;
- каждая клетка хорошо читается и по ней удобно водить мышью.

### 2. Main Canvas

Основная область редактора - это большая сетка из квадратов.

Требования:

- четкие границы клеток;
- визуально крупные "пиксели";
- удобно рисовать мышью;
- понятно, какие клетки уже заняты;
- видно, где линия начинается;
- видно, какие клетки выключены.

## Editing Modes

### 1. Paint Line Mode

Пользователь выбирает цвет и рисует линию по клеткам.

Ожидаемое поведение:

- нажал кнопку мыши;
- повел по клеткам;
- все клетки, через которые проходит курсор, закрашиваются выбранным цветом;
- линия рисуется непрерывно, как в редакторе пикселей;
- рисование должно быть комфортным, без необходимости кликать по одной клетке.

Нужно продумать два возможных поведения:

1. Свободное paint-поведение по клеткам.
2. Более строгий режим, где линия продолжается только в соседние клетки.

Для V1 лучше начать с простого и понятного варианта:

- рисование по задетым клеткам;
- позже добавить валидацию маршрута.

### 2. Inactive Cell Mode

Нужен специальный режим, который не связан с цветом линии.

Смысл:

- клетка не принадлежит активному игровому полю;
- она должна быть визуально помечена особым знаком.

Визуальное предложение:

- клетка не закрашивается цветом линии;
- поверх клетки ставится знак "X" или перекрестье;
- знак расположен по центру;
- неактивная клетка должна сразу отличаться от обычной пустой клетки.

Важно:

- inactive - это отдельное состояние клетки;
- оно не должно путаться с обычным цветом.

### 3. Start Marker Mode

Нужен способ помечать начало линии.

Смысл:

- вся линия имеет один цвет;
- у линии есть специальная стартовая клетка;
- стартовая клетка должна быть помечена отдельным значком.

Возможные варианты значка:

- маленький кружок;
- точка;
- стрелочный бейдж;
- специальный маркер в углу или центре клетки.

Для V1 лучше выбрать простой и читаемый вариант:

- маленький контрастный значок внутри клетки.

Важно:

- маркер начала не должен ломать основной цвет линии;
- он должен быть дополнительным overlay-слоем.

### 4. Select/Edit Mode

Нужен отдельный режим выбора и правки.

Он понадобится для:

- посмотреть, что уже нарисовано;
- выбрать клетку;
- поменять ее состояние;
- убрать ошибку;
- в будущем смотреть свойства линии или клетки.

Для V1 достаточно минимального поведения:

- выбор клетки;
- возможность стереть содержимое;
- возможность переключить состояние клетки.

## Color System

### Preset Palette

Нужна стартовая палитра предустановленных цветов, как в paint.

Например:

- red
- blue
- green
- yellow
- orange
- purple
- cyan
- brown
- black
- white or light gray for utility usage

Палитра должна быть:

- постоянно видна;
- быстро доступна;
- удобна для переключения.

### Custom Colors

Нужен color picker.

Он нужен для двух сценариев:

- добавить новый цвет;
- переопределить один из существующих.

Требования:

- можно выбрать кастомный цвет;
- можно сохранить его в текущую палитру;
- редактор должен помнить, какой цвет выбран сейчас.

## Core UX Requirements

### Must Have

- поле задается числами;
- поле строится кнопкой `Generate`;
- большие квадратные клетки;
- рисование мышью протягиванием;
- выбор текущего цвета;
- пометка inactive-клеток через отдельный режим;
- пометка начала линии через отдельный режим;
- базовое стирание.

### Nice to Have Later

- undo/redo;
- zoom;
- pan;
- show/hide grid;
- export/import;
- line validation;
- auto-detect connected regions by color;
- выделение всей линии одного цвета;
- редактирование уже созданной линии как объекта, а не только как пикселей.

## Data Thinking For Now

На этом этапе не фиксируем окончательный формат файла.

Вместо этого считаем, что редактор внутри себя должен уметь хранить хотя бы такие сущности:

- размер поля;
- состояние каждой клетки;
- цвет клетки, если это часть линии;
- признак inactive;
- признак start marker;

То есть пока мыслим не "как будет выглядеть JSON", а "какие состояния обязательно должны существовать в редакторе".

Минимальная модель состояния редактора:

- `width`
- `height`
- `cells`

У каждой клетки потенциально есть:

- `paintColor` or empty
- `inactive: true/false`
- `startMarker: true/false`

Позже уже можно будет понять:

- достаточно ли cell-based модели;
- или надо переходить к более строгой модели lines-as-objects.

## Important Open Design Question

Нужно будет отдельно решить, что является первичным:

1. Клетки, просто закрашенные цветом.
2. Полноценные линии как отдельные сущности.

Для первого прототипа редактора разумно начать с cell-paint модели, потому что:

- ее очень быстро сделать;
- она понятна визуально;
- по ней легко тестировать UX;
- она сразу покажет, удобно ли вообще рисовать уровни таким способом.

Если окажется, что этого недостаточно, следующим шагом можно перейти к гибридной модели:

- пользователь рисует по клеткам;
- редактор поверх этого собирает линии как объекты.

## Proposed Build Stages

### Stage 1. Visual Grid Prototype

Сделать пустой редактор с:

- width/height inputs;
- кнопкой `Generate`;
- отрисовкой большой квадратной сетки.

Цель:

- проверить базовый layout;
- проверить размер клеток;
- проверить, комфортно ли поле выглядит для 10x10, 20x20 и других размеров.

### Stage 2. Basic Paint Tools

Добавить:

- palette;
- current color;
- paint mode;
- drag painting;
- eraser.

Цель:

- понять, удобно ли рисовать линии руками.

### Stage 3. Special Markup Modes

Добавить:

- inactive mode;
- start marker mode;
- select mode.

Цель:

- понять, хватает ли такой модели для разметки головоломки.

### Stage 4. Internal State Inspection

Добавить простую debug-панель или raw preview состояния.

Например:

- временный JSON preview;
- список цветов;
- количество inactive cells;
- список cells with start markers.

Цель:

- посмотреть, какие данные реально получаются из редактора;
- уже на этом основании принять решение о формате сохранения.

### Stage 5. Save Format Decision

Только после ручного использования редактора решить:

- сохраняем ли все как клеточную карту;
- собираем ли линии по цветам;
- нужен ли двухслойный формат;
- нужен ли отдельный формат для runtime и отдельный для authoring.

## Recommended Technical Direction For Prototype

Для первого эксперимента лучше всего подойдет:

- `Flutter`-приложение;
- кастомный grid editor;
- акцент на ручное рисование;
- простое внутреннее состояние;
- без раннего усложнения логикой маршрутов.

Почему:

- `Flutter` хорошо подходит для tool-like UI: панели, кнопки, формы, color picker, режимы;
- один стек удобен для desktop-first разработки с возможностью позже проверить mobile;
- самый быстрый цикл изменений;
- удобно тестировать UX;
- легко потом добавить сохранение;
- легко потом ужесточить правила.

## Why Flutter Now

`Flutter` выбираем не потому, что `libGDX` не может это сделать, а потому что для первой версии редактора нам важнее удобство UI-разработки.

Что особенно хорошо ложится на `Flutter`:

- поля ввода размеров;
- кнопка `Generate`;
- панели инструментов;
- палитра;
- выбор текущего режима;
- color picker;
- desktop-like layout;
- drag interaction по сетке.

Что это нам дает:

- быстрее собрать первый рабочий редактор;
- быстрее менять UX по ходу проб;
- проще дойти до usable prototype до того, как мы зацементируем формат хранения.

## Flutter V1 Structure

Предварительно мыслим редактор как несколько простых UI-блоков:

### 1. Top Controls

- input `width`
- input `height`
- button `Generate`

### 2. Left Or Top Toolbar

- `Paint`
- `Inactive`
- `Start Marker`
- `Erase`
- `Select`

### 3. Palette Area

- набор preset colors
- текущий выбранный цвет
- кнопка открытия color picker
- возможность добавить или заменить цвет в палитре

### 4. Main Grid Canvas

- большая клеточная область;
- отрисовка крупных квадратов;
- drag-paint по клеткам;
- overlays для inactive и start marker.

### 5. Debug / State Preview

- временная панель состояния;
- позже поможет понять, как естественно сохранять результат.

## Definition Of Success For V1

Первый прототип успешен, если мы можем:

1. Задать размер поля.
2. Сгенерировать сетку.
3. Выбрать цвет.
4. Протянуть мышью цветную линию по клеткам.
5. Отметить некоторые клетки как inactive.
6. Поставить start marker.
7. Стереть ошибку.
8. Визуально понять, удобно ли таким способом собирать уровень.

Если это работает, значит мы уже можем принимать решение о формате хранения на основе практики, а не догадки.
