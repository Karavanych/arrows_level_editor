# Arrows Level Editor - Plan v0.2

## Goal

Сначала делаем удобный визуальный редактор уровня.
Формат сохранения не фиксируем заранее: сначала нужно понять, что именно мы рисуем, какие режимы реально удобны, и какие данные рождаются естественно из редактора.

Приоритет:

1. Быстро создать поле нужного размера.
2. Удобно рисовать линии цветами.
3. Удобно отмечать неактивные клетки.
4. Удобно видеть начало линии.
5. После этого зафиксировать editor-first формат сохранения.
6. Потом добавить save/load с валидацией.

## Main Product Decision

Первый шаг - не runtime, а standalone editor-first prototype.

Технологическое решение для первого прототипа:

- делаем редактор на `Flutter`;
- целимся в запуск на macOS и Windows;
- дополнительно оставляем возможность позже запускать тот же редактор на телефоне;
- `libGDX` сейчас не берем как основной стек редактора, потому что на этом этапе важнее скорость сборки tool-like UI, чем близость к игровому runtime.

Текущее состояние проекта:

- `Flutter`-прототип редактора уже собран;
- есть grid editor;
- есть palette;
- есть инструменты `Paint`, `Inactive`, `Start Marker`, `Erase`;
- есть pan / zoom;
- есть `undo/redo`;
- теперь следующий основной этап - формат сохранения и правила save validation.

Редактор должен быть похож на простой paint-like grid editor:

- есть крупная квадратная сетка;
- есть размеры поля;
- есть кнопка генерации поля;
- есть палитра цветов;
- есть режим рисования линий;
- есть режим пометки неактивных клеток;
- есть видимый маркер начала линии.

## Editor V1 Scope

### 1. Field Generation

Пользователь задает размер поля числами:

- width
- height

Примеры:

- 10x10
- 20x20
- любой произвольный размер в разумных пределах

Рядом с полями размеров должна быть кнопка:

- `Generate`

Поведение:

- при вводе новых чисел поле само не ломается мгновенно;
- по нажатию `Generate` создается или пересоздается новое поле;
- сетка визуально состоит из больших квадратных клеток;
- каждая клетка хорошо читается и по ней удобно водить мышью.

### 2. Main Canvas

Основная область редактора - это большая сетка из квадратов.

Требования:

- четкие границы клеток;
- визуально крупные "пиксели";
- удобно рисовать мышью;
- понятно, какие клетки уже заняты;
- видно, где линия начинается;
- видно, какие клетки выключены.

## Editing Modes

### 1. Paint Line Mode

Пользователь выбирает цвет и рисует линию по клеткам.

Ожидаемое поведение:

- нажал кнопку мыши;
- повел по клеткам;
- все клетки, через которые проходит курсор, закрашиваются выбранным цветом;
- линия рисуется непрерывно, как в редакторе пикселей;
- рисование должно быть комфортным, без необходимости кликать по одной клетке.

Нужно продумать два возможных поведения:

1. Свободное paint-поведение по клеткам.
2. Более строгий режим, где линия продолжается только в соседние клетки.

Для V1 лучше начать с простого и понятного варианта:

- рисование по задетым клеткам;
- позже добавить валидацию маршрута.

### 2. Inactive Cell Mode

Нужен специальный режим, который не связан с цветом линии.

Смысл:

- клетка не принадлежит активному игровому полю;
- она должна быть визуально помечена особым знаком.

Визуальное предложение:

- клетка не закрашивается цветом линии;
- поверх клетки ставится знак "X" или перекрестье;
- знак расположен по центру;
- неактивная клетка должна сразу отличаться от обычной пустой клетки.

Важно:

- inactive - это отдельное состояние клетки;
- оно не должно путаться с обычным цветом.

### 3. Start Marker Mode

Нужен способ помечать начало линии.

Смысл:

- вся линия имеет один цвет;
- у линии есть специальная стартовая клетка;
- стартовая клетка должна быть помечена отдельным значком.

Возможные варианты значка:

- маленький кружок;
- точка;
- стрелочный бейдж;
- специальный маркер в углу или центре клетки.

Для V1 лучше выбрать простой и читаемый вариант:

- маленький контрастный значок внутри клетки.

Важно:

- маркер начала не должен ломать основной цвет линии;
- он должен быть дополнительным overlay-слоем.

## Color System

### Preset Palette

Нужна фиксированная палитра предустановленных цветов, как в paint.

Текущее решение:

- палитра фиксированная;
- 24 цвета;
- layout: `4 x 6`;
- клик по цвету сразу включает режим `Paint`.

Палитра должна быть:

- постоянно видна;
- быстро доступна;
- удобна для переключения.

## Core UX Requirements

### Must Have

- поле задается числами;
- поле строится кнопкой `Generate`;
- большие квадратные клетки;
- рисование мышью протягиванием;
- очистка правой кнопкой мыши с drag-поведением;
- `Ctrl/Cmd + drag` двигает поле;
- `undo/redo`;
- выбор текущего цвета;
- пометка inactive-клеток через отдельный режим;
- пометка начала линии через отдельный режим;
- базовое стирание.

### Nice to Have Later

- show/hide grid;
- export/import;
- line validation;
- auto-detect connected regions by color;
- выделение всей линии одного цвета;
- редактирование уже созданной линии как объекта, а не только как пикселей.

## Current Editing Rules

Уже приняты и важны для формата такие правила:

- правая кнопка мыши всегда полностью очищает клетку;
- очистка удаляет:
  - цвет
  - inactive
  - start marker
- `Paint` не рисует поверх уже занятой клетки;
- `Inactive` не ставится поверх уже занятой клетки;
- чтобы заменить содержимое клетки, сначала нужно явно стереть старое;
- `Inactive` по смыслу является отдельным конечным состоянием клетки;
- точный клик по уже закрашенной клетке в режиме `Paint` выбирает этот цвет;
- в остальных режимах такого color-pick из поля нет;
- при сохранении пустых клеток быть не должно.

## Save Format Decision

Формат сохранения фиксируем как свой контейнерный формат уровня/пака.

Рабочее имя:

- `*.alevelpack`

Это не runtime-формат игры и не внутренний state редактора.
Это authoring/storage format для хранения одного или нескольких уровней.

## Why This Format

Выбрали свой контейнерный формат, потому что нам одновременно нужно:

- хранить поле как цветовую карту;
- хранить стартовые точки отдельно;
- уметь хранить несколько уровней;
- иметь один файл снаружи, а не пару разрозненных файлов;
- оставить путь к будущему расширению формата без переделки базовой идеи.

## ALevelPack v1

### Outer Format

`alevelpack` v1 - это один контейнерный файл с собственным расширением.

Практическая реализация:

- обычный zip-контейнер;
- собственное расширение файла, например `main.alevelpack`.

Это дает:

- один файл для пользователя;
- простую упаковку/распаковку;
- возможность хранить несколько уровней;
- возможность расширять формат без перехода на сырой бинарный парсер.

### Internal Structure

Предлагаемая структура контейнера:

```text
manifest.json
palette.json
levels/
  level_001/
    board.png
    meta.json
  level_002/
    board.png
    meta.json
```

### manifest.json

Содержит:

- идентификатор формата;
- версию;
- имя пака;
- список уровней;
- порядок уровней.

Черновой пример:

```json
{
  "format": "arrows-level-pack",
  "version": 1,
  "name": "main-pack",
  "levels": [
    { "id": "level_001", "path": "levels/level_001" },
    { "id": "level_002", "path": "levels/level_002" }
  ]
}
```

### palette.json

Содержит:

- список допустимых цветов;
- зарезервированный цвет для inactive.

Черновой пример:

```json
{
  "inactiveColor": "#FFFFFF",
  "colors": [
    "#D32F2F",
    "#E53935",
    "#F06292"
  ]
}
```

### board.png

Это основная карта уровня.

Правила:

- размер изображения = размер поля;
- `1 пиксель = 1 клетка`;
- цвет пикселя = состояние клетки;
- обычные цвета = части линий;
- специальный белый цвет = `inactive`;
- пустых клеток в сохраненном уровне быть не должно.

То есть после сохранения весь board должен быть полностью заполнен пикселями допустимых цветов.

### meta.json

Содержит данные, которые не должны кодироваться цветом.

На первом этапе:

- `startPoints`

Черновой пример:

```json
{
  "startPoints": [
    { "x": 2, "y": 5 },
    { "x": 8, "y": 1 }
  ]
}
```

## Data Semantics

### Field Encoding

- любой обычный цвет = обычная игровая клетка / часть линии;
- `#FFFFFF` = `inactive`;
- “пустой клетки” в сохраненном уровне не существует.

### Start Points

Стартовые точки храним отдельно от bitmap.

Причина:

- не хотим смешивать цвет клетки и служебный смысл старта;
- не хотим заводить специальные “стартовые оттенки” цветов;
- стартовые точки sparse по природе, для них metadata естественнее.

### Multi-Level Storage

В одном `alevelpack` можно хранить несколько уровней.

Это пригодится для:

- набора уровней;
- тестовых паков;
- редакторной работы пачками;
- будущей группировки уровней по темам или мирам.

## Editor File Workflow

Редактор работает с одним текущим `alevelpack` файлом.

### File Creation And Update

- если файла сохранения еще нет, редактор должен создать новый `alevelpack`;
- если файл уже существует, редактор должен дописывать / обновлять данные в этом же файле;
- базовый сценарий пользователя: открыть существующий пак или создать новый при первом сохранении.

### Default Storage Location

Файл нужно хранить в стандартной пользовательской папке приложения для Flutter desktop/mobile приложения.

То есть:

- не во временной папке;
- не рядом с executable;
- а в нормальной app-specific documents / application support папке платформы.

Точное техническое API и конкретный platform path можно выбрать при реализации, но продуктовое требование уже фиксируем:

- редактор должен иметь одно стандартное место хранения pack-файлов.

### Save And Open Actions

В редакторе нужны явные действия:

- `Save`
- `Open`

Базовая логика:

- `Open` открывает текущий `alevelpack` файл и загружает список уровней из него;
- `Save` сохраняет текущий уровень и текущее состояние пака;
- если файла еще нет, `Save` создает новый файл;
- если файл есть, `Save` обновляет существующий пак.

## Level List In The Right Panel

Правая служебная панель должна использоваться как список уровней текущего открытого файла.

Что там должно быть:

- список уровней из текущего `alevelpack`;
- возможность кликнуть по уровню и переключиться на него;
- визуально понятный текущий выбранный уровень.

Это нужно потому, что:

- один pack может хранить несколько уровней;
- пользователь должен быстро переходить между уровнями внутри одного файла.

## Unsaved Changes Flow

При переключении уровня нужно учитывать несохраненные изменения.

Правило:

- если пользователь пытается переключиться на другой уровень, а в текущем уровне есть несохраненные изменения, редактор должен сначала предложить сохранить изменения.

Базовый flow:

- попытка перейти на другой уровень;
- если текущий уровень dirty, показываем вопрос о сохранении;
- только после этого продолжаем переключение уровня.

Точный состав кнопок диалога можно отдельно утвердить при реализации, но само правило уже фиксируем:

- level switch не должен молча терять несохраненные изменения.

## Save Validation Direction

Перед сохранением нужен явный этап проверки.

Полный список ошибок еще надо отдельно утвердить, но уже сейчас фиксируем несколько базовых направлений:

- нельзя сохранять уровень, если есть пустые клетки;
- нельзя сохранять уровень, если встречается цвет вне допустимой палитры;
- нельзя сохранять уровень, если `startPoints` указывают на невалидные клетки;
- позже отдельно обсудим логические ошибки уровня и жесткость этих проверок.

### Approved Rule 1. Empty Cells

Пустые клетки являются save-blocking ошибкой.

Поведение редактора:

- при попытке сохранить уровень с пустыми клетками все пустые клетки должны мигнуть красным;
- базовое поведение мигания: 2 раза;
- это принимаем как общий визуальный паттерн для editor errors: ошибочные клетки или другие проблемные элементы подсвечиваем через мигание.

После этого редактор должен задать явный вопрос пользователю:

- заполнить ли все пустые клетки как `inactive`?

Варианты ответа:

- `Да`
- `Отмена`

Если пользователь выбирает `Да`:

- все пустые клетки автоматически заполняются как `inactive`;
- после этого редактор повторяет сохранение.

Если пользователь выбирает `Отмена`:

- сохранение отменяется;
- пользователь возвращается к редактированию уровня без автозаполнения.

### Approved Rule 2. No Single-Cell Color Islands

Одиночная клетка обычного цвета является save-blocking ошибкой.

Минимальный допустимый размер цветовой группы:

- 2 клетки.

Связность клеток считаем только по ортогональным соседям:

- слева
- справа
- сверху
- снизу

Диагональ не считается соединением.

Следствие:

- если у цветной клетки нет ни одного соседа того же цвета по 4-связности, такая клетка считается ошибкой;
- одиночные цветные клетки нельзя сохранять в уровне.

UI-поведение для этой ошибки:

- все ошибочные одиночные клетки должны мигнуть красным;
- базовое поведение мигания: 2 раза;
- это подчиняется общему правилу editor errors через мигание проблемных элементов.

### Approved Rule 3. Every Color-Connected Line Must Have A Start

Для каждой отдельной линии должно быть указано начало.

Линией на этом этапе считаем отдельную связную компоненту клеток одного цвета.

Правила связности:

- соединение считается только по ортогональным соседям;
- используются только связи:
  - слева
  - справа
  - сверху
  - снизу
- диагональ соединением не считается.

Важно:

- в уровне допускаются несколько разных линий одного и того же цвета;
- проверка должна идти не по цвету целиком, а по каждой отдельной связной компоненте этого цвета.

Следствие:

- у каждой color-connected линии должен быть start marker;
- если у отдельной линии начала нет, это save-blocking ошибка.

UI-поведение для этой ошибки:

- все линии без начала должны мигнуть красным;
- базовое поведение мигания: 2 раза;
- после этого редактор должен задать вопрос, поставить ли начало автоматически.

Варианты ответа:

- `Да`
- `Отмена`

Если пользователь выбирает `Да`:

- редактор автоматически ищет стартовую клетку для каждой линии без начала;
- после этого редактор повторяет сохранение.

Если пользователь выбирает `Отмена`:

- сохранение отменяется;
- пользователь возвращается к редактированию уровня без auto-fix.

### Temporary Auto-Start Rule

Пока используем упрощенную временную логику определения старта:

- для линии ищем любую клетку, у которой ровно 1 соединение с клеткой того же цвета по 4-связности;
- такую клетку считаем началом линии;
- если подходящих клеток несколько, можно брать любую;
- эта логика временная и позже будет заменена или усложнена.

### Dedicated Line Analysis Class

Логику такого анализа нужно вынести в отдельный класс.

Этот класс должен отвечать минимум за:

- поиск color-connected линий;
- определение start candidate для линии;
- проверку, что у линии есть начало.

Именно этот класс позже будет расширяться для:

- более умного определения начала;
- более строгой save validation;
- отдельной проверки играбельности поля.

То есть архитектурно сразу закладываем отдельную точку роста, а не размазываем эту логику по save-коду.

## What We Save Versus What We Derive

Сохраняем прямо:

- размер поля;
- board bitmap;
- start points;
- palette metadata;
- manifest / level ordering.

Не сохраняем пока как первичную сущность:

- route objects;
- line path objects;
- head direction objects;
- runtime graph.

Это все потом может быть вычислено или экспортировано отдельно.

## Important Open Design Question

Нужно будет отдельно решить, что именно считается ошибкой уровня при сохранении.

То есть различать:

- технически невалидный файл;
- редакторное предупреждение;
- логически плохой, но сохраняемый уровень.

Первое уже зафиксированное правило:

- пустые клетки - это save-blocking ошибка с возможностью явного auto-fix через массовое заполнение `inactive`.

Второе уже зафиксированное правило:

- одиночная цветная клетка недопустима;
- минимальная цветовая группа = 2 клетки;
- связность считается только по 4 соседям;
- диагональ соединением не считается.

Третье уже зафиксированное правило:

- каждая отдельная связная линия одного цвета должна иметь начало;
- несколько линий одного цвета допускаются;
- проверка идет по связным компонентам, а не по цвету в целом;
- при отсутствии начала допускается временный auto-fix через поиск клетки с одной связью.

Остальной список остается следующим большим блоком обсуждения после фиксации структуры `alevelpack`.

## Proposed Build Stages

### Stage 1. Visual Grid Prototype

Сделать пустой редактор с:

- width/height inputs;
- кнопкой `Generate`;
- отрисовкой большой квадратной сетки.

Цель:

- проверить базовый layout;

## Recommended Technical Direction For Prototype

Для первого эксперимента лучше всего подойдет:

- `Flutter`-приложение;
- кастомный grid editor;
- акцент на ручное рисование;
- editor-first internal state;
- authoring container format поверх этого state.

Почему:

- `Flutter` хорошо подходит для tool-like UI: панели, кнопки, формы, color picker, режимы;
- один стек удобен для desktop-first разработки с возможностью позже проверить mobile;
- самый быстрый цикл изменений;
- удобно тестировать UX;
- легко потом добавить сохранение;
- легко потом ужесточить правила.

## Why Flutter Now

`Flutter` выбираем не потому, что `libGDX` не может это сделать, а потому что для первой версии редактора нам важнее удобство UI-разработки.

Что особенно хорошо ложится на `Flutter`:

- поля ввода размеров;
- кнопка `Generate`;
- панели инструментов;
- палитра;
- выбор текущего режима;
- color picker;
- desktop-like layout;
- drag interaction по сетке.

Что это нам дает:

- быстрее собрать первый рабочий редактор;
- быстрее менять UX по ходу проб;
- проще дойти до usable prototype до того, как мы зацементируем формат хранения.

## Definition Of Success For V1

Первый прототип успешен, если мы можем:

2. Сгенерировать сетку.
